### 학습 목표

웹 애플리케이션 빌드 시, 정말 필요한 코드만 골라서 번들에 포함하고 나머지는 쳐내는 최적화 과정.

-   다양한 번들러가 어떻게 Tree Shaking을 구현하는지
-   다양한 번들러(Webpack, Rollup, esbuild, Turbopack)들 간에 어떤 차이가 있는지
-   단순히 파일을 합치는 도구를 넘어 어떤 조건에서 미사용 코드를 제거할 수 있는지 동작 원리 이해
-   모듈 레벨 vs AST-node 레벨 분석이 어떤 영향을 주는지
-   실제 프로젝트에서 번들 크기 최적화, 빌드 설정 조정은 어떻게 할지

### 트리셰이킹

현대 프론트엔드 번들링에서 매우 중요하고 필수적인 부분이다.
다양한 번들러들은 중점하는 게 달라 트리 셰이킹 구현 방식도 상이하다.

-   웹팩은 주로 FE 애플리케이션 번들링에 사용되며 정확성을 중시한다. 모듈 간 수준의 최적화에 초점을 둔다.
-   롤업은 주로 라이브러리 번들링에 사용된다. 하지만 최적화 효율성을 우선시한다. 예를 들어, AST 노드 단위 세밀한 수준에서 수행하여 일반적으로 결과물 번들 크기가 더 작다. 그러나 특정 극단적인 경우에는 실행 정확성에 대한 보장이 덜할 수 있다.

다양한 번들러의 트리셰이킹 원리와 그 차이점을 알아보자.

### 트리셰이킹이란?

애플리케이션이 나무라면,
실제 사용 소스코드와 라이브러리는 나무의 잎을 나타낸다.
죽은 코드는 가을에 떨어지는 낙엽을 상징한다.
죽은 잎을 제거하려면, 나무를 흔들어 떨어뜨려야 한다.

### 웹팩/Rspack에서의 트리 셰이킹

현재 Rspack(v1.4)은 웹팩과 동일한 트리 셰이킹 구현 방식을 다르고 있다.

웹팩의 트리셰이킹은 3부분으로 구성된다.

-   모듈 수준

    -   `optimization.sideEffects` 사용되지 않은 내보내기와 사이드 이펙트가 없는 모듈을 제거

        -   `import "./module";` 내보내기된 항목을 사용하지 않고 사이드 이펙트가 없으므로 안전 제거 가능
        -   `re-exports.js(배럴 파일)` 자체는 로컬 내보내기 항목이 없으며 사용되고 있고 사이드 이펙트가 없으므로 안전 제거 가능

        ```
        // index.js
        import { a } from "./re-exports";
        console.log(a);

        // re-exports.
        export * from "./module";
        // `index -(a)-> re-exports -(a)-> module`이 `index -(a)-> module`로 최적화되었으며, re-exports.js 자체는 로컬 내보내기가 없으며 사용되고 있고, 사이드 이펙트가 없으므로 re-exports.js는 안전 제거 가능

        // module.js
        export const a = 42;
        ```

    -   export-level: `optimization.providedExports` 및 `optimization.usedExports`를 사용하여 사용되지 않는 내보내기 항목 제거
