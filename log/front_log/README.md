### 필요성

더 나은 비즈니스 의사결정을 위해 그에 도움 될 데이터를 수집하기 위함.

### 유형

-   실제 사용자의 인터랙션 이벤트 로그 수집 (비즈니스적 가치 ⇒ 가설 검증, 의사결정)

### 무엇을 로깅해야 하는가

-   가장 어려운 부분.
-   사용자의 어떤 상호작용을 로그로 만들어 데이터화 해야하는지 결정

### 로그 레벨의 의미와 운영 기준

-   TRACE
    -   극세밀 디버깅 (거의 안 씀)
    -   극한 디버깅용. 거의 안 씀
-   DEBUG
    -   개발 중 흐름 확인 (개발환경 전용)
    -   내 디버깅용
-   INFO
    -   정상 동작 기록 (운영환경에 기록)
    -   서비스 정상 흐름 기록용
    -   사용자가 어떤 행동을 했는지는 꼭 남긴다.
        ```html
        logger.info("user {} logged in successfully", userId) logger.info("Payment completed:
        orderId={}", orderId)
        ```
-   WARN
    -   주의상황 경고 (운영 모니터링)
    -   위험 신호 감지용
    -   문제 징조. 심각 에러는 아니지만 조치가 필요한 상황
        ```html
        logger.warn("Disk usage high: {}%", diskUsage) logger.warn("Third-party API 응답 느림:
        responseTime={}ms", responseTime)
        ```
-   ERROR
    -   실패, 장애 기록 (운영 필수)
    -   시스템 실패 기록용 (스택트레이스 필수)
        ```html
        try { ... } catch (Exception e) { logger.error("DB 연결 실패", e) }
        ```
        나중에 어떤 에러가 어디서 났는지 찾을 때 필수

### 로그 작성 4대 원칙 (추적성, 보안, 성능)

-   로그 남길 때, `누구(User)`, `무엇(Order)` , `언제(When)` 을 꼭 남긴다.
-   `민감 데이터`는 절대 기록하지 않는다. (비번, 주민번호)
-   성능 고려: `for문`, `자주 호출 함수` 에는 debug 이상 쓰지 않는다.
-   `구조화된(JSON)` 로그로 남겨 분석 가능하게 한다. (Logstash 연동 대비)

### 작성/운영 심화 전략 7단계

1. 구조화된(JSON) 로그 작성
2. Context 정보 (userId, ip 등) 항상 포함 ⇒ 스프링부트 MDC로 Context 정보 삽입
3. Correlation ID 삽입 ⇒ 필터에서 Correlation ID 생성
4. 민감 데이터 완전 차단 ⇒ @JsonIgnore
5. 로그 레벨별 파일 분리 저장 ⇒ Logback
6. 로깅 비용 최적화 ⇒ `if (logger.isDebugEnabled())`로 불필요 연산 차단
7. 운영환경에서는 비동기 로깅 적용 ⇒ `AsyncAppender`로 비동기 로깅 구현

### 백엔드 비동기 로깅 필수 이유

-   서버 초당 수백~수천 요청 처리
-   동기 로그로 디스크 I/O 병목 발생
-   AsyncAppender로 메모리 버퍼 처리 후 비동기 기록
-   고성능 서버도 디스크는 병목되므로 초기 대비 필요

### 핵심

-   로그는 추적성, 보안, 성능, 운영 최적화 위한 필수 자산
-   개발 중 DEBUG, INFO 사용, 운영 중 INFO, WARN, ERROR 만 사용
-   JSON 구조화 + Context 삽입 + Correlation ID 기반 추적
-   서버는 반드시 비동기 로깅 적용

### X-Correlation-ID

상관관계 ID

여러 서비스, 구성요소를 거치며 이동할 때 사용되는 고유 식별자

문제 발생 시 이전 단계를 되짚어볼 수 있도록

여러 서비스, 로그 항목에서 트랜잭션 추적하는 고유 식별자

모든 관련 요청, 메시지, 응답을 연결하는 단일 태그 역할

-   작동 원리
    -   생성: 거래에 대한 고유 식별자 생성 (종종 uuid)
    -   전파: X-Correlation-ID 초기 요청 헤더에 포함, 모든 후속 요청이나 메시지와 함께 전달됨
    -   로깅: 관련 모든 서비스는 로그 메시지에 포함시켜 일관된 감사 추적

### X-Tracing-ID

추적 ID

상관관계 ID보다 발전된 형태

서로 다른 작업 간 관계 파악하는 더욱 정교한 추적 프레임워크

-   특징
    -   계층구조 (부모-자식 관계 가질 수 있는 스팬 포함)
    -   타이밍: 작업 발생 시간뿐 아니라 지속시간도 캡처
    -   자세한
