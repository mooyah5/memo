# JS Class & Mixin

JS는 ES2015(ES6)부터 클래스 문법이 도입되었다.

그러나, 실은 `클래스처럼` 동작하게 만들어주는 `편의문법(syntactical sugar)`이라는 점에 유의해야 한다.

내부적으로 기존 `prototype` 기반 상속구조 변환되기에,
사용자가 작성한 클래스는 결국
`new` 키워드와 함께 호출되기를 기다리는 js function으로 바뀌게 된다.
그럼에도 기존 prototype 기반 상속모델보다 훨씬 깔끔/명확하게 객체 상속구조를 정의할 수 있다.
메서드 내부 `this` 키워드는 항상 메서드를 담은 클래스의 인스턴스를 참조한다는 점도 유용하다.

### js Class 선언과 상속

ES2015 클래스 문법이 제공하는 `class`, `extends` 키워드를 이용하면 클래스 계통을 명확하게 표현할 수 있다.

새롭게 추가된 클래스 문법에서는 `생성자(constructor)`를 제외할 수 있고,
이 경우 상부 클래스의 생성자가 호출된다.

`new` 키워드로 만들어진 클래스의 인스턴스는
부모 생성자 호출을 통해 속성값을 지니게 된다.

생성자를 명시적으로 선언하고 싶다면 `전개 연산자(Spread Operator)` 구문으로
부모에 전달할 파라미터를 아래와 같이 편하게 작성 가능

```
class Duck extends Animal {
    constructor(...args) {
        super(...args)
    }
}
```

### 클래스 상속구조의 붕괴

`미키 마우스`를 만들려면 `Animal`에서 파상된 `Mouse` 클래스가 필요하다.
하지만 쥐는 날지 못한다. (`flyTo??`)
설계 논리가 붕괴되는 시점이다.

이를 해결하려면, `flyTo`를 제거하고 이를 사용하던 기존 모든 클래스에 하나씩 붙여나가야 한다.
잘못된 상속구조 설계로 인해 값비싼 비용을 지불하게 되었다.

설계자의 잘못일까? 쉽지는 않은 일이다.

코드 재사용을 목적으로 `상부 클래스 계통이 지나치게 많은 책임, 기능을 부여하면 미래 변화에 대응할 유연성을 잃게 된다.`

만약, 행위(기능)들이 클래스 계통에서 분리되어 있다면 어떨까?
새로운 행위가 추가되고 수정돼야 할 때, 전체 클래스 계통을 살필 필요가 없어진다.

### 행위의 분리 - 믹스인(Mixins)

필요 행위를 독립된 클래스로부터 얻어와 탑재할 수 있으면 훨씬 유연한 재사용 패턴이 된다.
개발 선배들은 이를 위한 해결책으로 `믹스인`을 남겨 두었다.

믹스인은 `특정 기능만을 담당하는 클래스`로,
단독 사용이 아닌, 클래스에 탑재되어 사용할 목적으로 작성된 `조각(클래스)`를 의미한다.

믹스인 클래스를 다른 클래스에 `담는 방법은 언어별로 다양`하다.
특히 `컴파일`언어와 `인터프리터`언어는 적용방식에 있어 큰 구조적 차이를 보인다.

js의 경우, `클래스 문법`, 기존 `prototype` 기반 모델, `Object.assign`로 객체에 직접 행위 붙이기 등이 있다.

GOF의 디자인 패턴 - `유연하고 재사용 가능한 설계를 정확하게 하기는 불가능하거나 매우 어려운 일`

최소한의 수정으로 재사용 되는 설계가 가장 훌륭한 설계일까?
믹스인은 설계 유연성을 높이는 주요 기법 중 하나다.

### 참고

1. 믹스인이 많아질수록 클래스 선언부에 기술하는 양이 많아지는데, 이때 `lodash`의 `compose` 기능을 이용하면 편리하게 조합할 수 있다.

```
import compose from 'lodash/fp/compose';

const behavios = compose(FlyToMixin, EatMixin, SwimAtMixin)(Animal)
class Duck extends behaviors {}
```

2. 믹스인으로 `장식자 패턴(Decorator)`을 구현할 수 있다.
