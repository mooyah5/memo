# Work Thred (워커 스레드)

### 목적
서버는 CPU 집약적 워크로드로 빠르게 과부하가 걸릴 수 있다.
- 예) 두 개의 엔드포인트.
    - 1. 단순 비 CPU 집약적 작업
    - 2. 10초 걸리는 복잡한 CPU 집약적 작업
    2번 수행하느라 1번을 미룰 수 있음 ㅠ
    => Node.js의 황금률, 즉 이벤트 루프를 차단하지 않는다는 원칙을 위반해서 발생.

### 싱글 스레드 아키텍처
노드는 싱글스레드 아키텍처 기반이라, CPU 집약적 작업에 적합하지 않다.

- 노드가 시작되면 실행되는 과정
    - 하나의 프로세스: 어디서든 접근 가능한 전역 객체이자, 그 순간 실행되고 있는 것들의 정보를 가진 프로세스
    - 하나의 스레드: 단일 스레드는 주어진 프로세스에서 한 번에 하나의 명령만 실행됨
    - 하나의 이벤트 루프: js가 단일스레드임에도 불구하고, 언제든 가능한 callback, promise, async/await로 시스템 커널에 작업을 offload하게 함. 이로서 노드가 비동기식, 비차단 I/O의 특성을 가짐.
    - 하나의 js 엔진 인스턴스: js 코드를 실행하는 컴퓨터 프로그램
    - 하나의 노드js 인스턴스: 노드js 코드를 실행하는 컴퓨터 프로그램

즉, 노드는 하나의 코드, 하나의 실행. 병렬로 실행되지 않는다.
`동시성 문제` 걱정이 없다는 점은 유용하나, 
인 메모리에서 발생하는 대규모 데이터 세트의 복잡한 계산 같은 CPU 자원을 많이 사용하는 코드가 있으면, 이 코드가 다른 프로세스의 실행을 차단할 수도 있다.


##### Node.js v10.5.0에서는 `worker_threads` 모듈로 워커 스레드를 도입했다. (v12부터 안정화)


### 워커 스레드 이해 전제 조건

#### 동시성 vs 병렬성
- 동시성: 한 번에 여러 작업을 `싱글 코어`에서 처리 (Concurrency)
- 병렬성: 한 번에 여러 작업을 `멀티 코어`에서 처리 (Parallelism)

동시성은 병렬성보다 오래 걸릴 수 있다.
네트워크 통신, 파일/DB 읽기쓰기 같이 기다리는 `I/O 작업는 동시성 모델이 유용하다.`

그러나, `CPU 집약적 작업에는 적합하지 않을 수 있다.` 멀티 스레드/코어를 활용해 CPU 집약적 작업은 워커 스레드에 위임하고, 비 CPU 집약적 작업은 메인 스레드에서 실행할 수 있었다.


```
Nodejs에서 I/O 작업은 별도 처리되고,
완료 시 이벤트 루프가 해당 I/O 작업과 연관된 콜백을
마이크로태스크 큐에 추가한다.

메인 스레드의 콜 스택이 비워지면,
콜백이 콜 스택에 푸시된 후 실행된다.

중요한 건,
I/O 작업 관련 콜백함수 자체는 병렬로 실행되지 않지만,
I/O 작업 자체는 스레드 도움을 받아 병렬 처리된다는 점이다.
I/O 작업이 완료되면 관련 콜백은 메인 스레드에서 실행된다.
```

#### 프로세스와 스레드
- 프로세스
    OS에서 실행 중인 프로그램.
    각각 `자체 메모리`가 있고, 실행 중인 `다른 프로그램 메모리를 보거나 접근할 수는 없다.`
    한 프로세스는 한 번에 한 작업만 수행한다.
    - 싱글 코어 시스템은 OS가 일정 간격으로 `프로세스를 전환`하면서 여러 프로세스가 동시 실행된다.
    - 멀티 코어 시스템은 OS가 각 코어에서 개별 프로세스를 동시 실행하도록 스케줄링한다.

- 스레드
    한 번에 한 작업을 실행한다.
    프로세스와 다르게, 자체 메모리가 없다. (대신 `프로세스 메모리 내에 존재`)
    한 프로세스가 만들어지면, 여러 스레드가 코드를 병렬 실행할 수 있다.
    `스레드 간에는 메시지 전달, 프로세스 메모리의 데이터 공유로 서로 통신`도 된다.
    - 그래서 스레드는 프로세스보다 가볍다. (새로운 스레드 만들 때 OS로부터 추가 메모리 할당 필요 없음)


## 워커스레드란?
워커스레드 모듈을 통해 CPU 집약적 작업을 다른 스레드로 위임해서
메인 스레드 차단을 방지할 수 있다.
- 예시 코드
    - `worker.js`: CPU 집약적 작업 수행
    - `parent.js`: `worker-threads` 모듈을 사용하여 스레드를 초기화하고, `worker.js` 작업 시작

```
// parent.js

import { Worker } from "work_threads"

const worker = new Worker("./worker.js", { workerData: { num: 5 }}) // Worker 실행하여 새로운 스레드 생성. worker.js 파일 코드가 다른 코어 스레드에서 실행되기 시작한다.

//// on("message") 메서드
// worker 인스턴스에 이벤트를 추가하고 메시지 이벤트를 수신하도록 설정.
// 해당 파일에서 결과 포함 메시지를 수신하면 메서드 콜백에 매개변수로 전달되고,
// 콜백은 CPU 집약적 작업 결과가 포함된 응답을 사용자에게 반환함.

worker.on("message", res => {
    console.log('Square of 5 is: ', res)
})

worker.on("error", msg => { // 오류 이벤트 수신
    console.log(msg)
})

console.log("mooyaho")

```

```
// worker.js

import { parentPort, workerData } from "worker_threads" 

// parentPort: 메인 스레드로 메시지를 전송할 수 있는 메서드 제공
// workerData: 스레드가 초기화될 때 메인 스레드에서 전달된 데이터 포함

parentPort.postMessage(workerData.num * workerData.num) 
```


## 노드 내부
Node.js 프로세스가 시작되면
하나의 프로세스, 하나의 스레드, 하나의 이벤트 루프, 하나의 V8 엔진 인스턴스, 하나의 Node.js 인스턴스로 시작된다.




##### CPU 작업과 I/O 작업 구분
- 노드는 병렬로 실행되지 않는다.
- 오직 I/O 작업만 비동기식으로 실행되므로, 병렬 실행된다.
- 그래서 워커 스레드는 I/O 집약적인 일에는 효과적이지 않다. (비동기적 I/O 작업이 워커보다 더 효율적)

`워커의 중요한 목표는 I/O작업이 아닌, CPU 집약적 작업의 퍼포먼스 향상이다.`
